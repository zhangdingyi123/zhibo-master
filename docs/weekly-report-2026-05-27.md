## 五、支付链路：现状与待解决问题

### 5.1 已实现（Mock）

- 成交后 `OrderService.CreateFromSettledSession` 自动生成 `pending_pay` 订单
- 用户端：`GET /orders`、`GET /orders/:id`、`POST /orders/:id/mock-pay`
- 前端：结果页「去支付」→ 订单支付页 → 模拟支付成功

### 5.2 仍存在的问题（产品 / 技术债）

| 问题 | 影响 | 建议方向 |
|------|------|----------|
| **仅 Mock 支付** | 无法对接真实资金与对账，演示外不可用 | 接入微信/支付宝沙箱或统一支付网关 |
| **无支付超时** | `pending_pay` 订单可长期悬挂，库存/履约语义不清 | 增加 `pay_expire_at` + 定时关单或释放 |
| **无支付回调幂等** | 真实渠道需处理重复通知 | 以 `order_no` + 状态机做幂等更新 |
| **订单列表无商品 JOIN** | 前端需按 `sessionId` 二次请求详情补全商品（N+1） | 后端 `ListForBuyer` 返回 `product` 摘要 |
| **中标后引导弱** | 成交 1.5s 自动跳结果页，部分用户来不及看清终态 | Toast 提供「稍后查看 / 立即支付」选项 |
| **未登录中标** | 结果页提示登录，但无订单与身份的临时绑定说明 | 登录后按 `sessionId` 拉单并校验 `buyerId` |
| **无退款/取消** | 异常流（拍错、主播取消已支付）未覆盖 | 扩展 `cancelled`/`refunded` 状态与 admin 操作 |
| **管理端与用户端订单割裂** | 主播侧仅看 seller 订单，买家侧无物流/收货 | 按课题范围决定是否补充履约字段 |

### 5.3 本周已缓解的前端支付体验问题

- 订单列表仅显示订单号 → **已展示商品图、名称、待支付「去支付」提示**
- 支付按钮文案「模拟支付」易误解 → **已加演示环境说明条，按钮改为「确认模拟支付」**
- 待支付订单不易发现 → **底部 Tab 订单角标（轮询 `pending_pay` 数量）**
